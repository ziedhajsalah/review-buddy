Review Buddy

A pull request isn't a wall of diffs — it's a story. An agent reads the changes, writes a Prologue, and segments the work into risk-rated Chapters you walk through in the browser.

v0.3.1 · Phase 1 complete Claude Code · Cursor · Copilot · Codex MCP tool + PreToolUse hook reference-not-reproduce
01 — THE IDEA

From a diff dump to a guided reading

The central abstraction is the Chapter: a thematic grouping of related changes — one logical sub-problem — carrying its own risk rating, stats, file set, and an AI-written explanation. Above them sits the Prologue, an orientation summary of the whole PR. All agent prose renders as markdown (GFM, no raw HTML).

02 — THE FLOW

How a review reaches the browser

The agent calls our own MCP tool. A PreToolUse hook intercepts it — chosen over PermissionRequest because it always fires, even in auto-accept modes — captures the real diff, opens the UI, and blocks the turn until you're done.

The agent supplies judgment and structure; git supplies the bytes. The hook is the conductor.

The hook never sees your command. The agent's submit_review carries a source — { type: "worktree" }, { type: "pr", ref }, or { type: "branch", ref } — so the hook re-captures the same diff the agent read. For a PR it runs gh pr diff <ref> against the PR's real bytes; without source the viewer would render empty. You never have to check the branch out.

No hook? Standalone mode. In Cursor / VS Code Copilot / Codex there is no PreToolUse — the SAME MCP server runs with the --standalone flag, and submit_review itself performs the capture → merge → serve steps, returning the viewer URL immediately (detached) so harness tool-call timeouts (Cursor ~5 min hard, Copilot unconfigurable) can't kill a human review; Codex can opt into --standalone=blocking paired with tool_timeout_sec. The viewer lives inside the long-lived MCP process, and shutdown is wired to client disconnect + SIGINT/SIGTERM so an editor reload can't orphan the port. See docs/HARNESSES.md.

03 — THREE DATA SOURCES

The agent JSON is only the narrative layer

The UI needs three sources merged together. The agent never reproduces the diff — it references it.

SOURCE A

Agent review JSON

Prologue, chapters, risk, descriptions, chapter→file grouping, and hunk anchors.

From the submit_review tool call.

SOURCE B

git diff + PR metadata

Authoritative hunk content + line numbers, original PR description, author, base/head, CI.

Run by the hook: git diff / gh.

SOURCE C

Full file contents

Bytes around the diff — for "expand full file" and word-level highlighting.

Fetched on demand via git show (PR-aware).

04 — THE LOAD-BEARING RULE

Reference, don't reproduce

The agent emits hunk anchors — old_start / new_start — never diff line content. The tool parses the real git diff and maps each authoritative hunk onto a chapter by path + anchor, then recomputes every stat.

Why it's load-bearing. LLMs alter whitespace, drop lines, and miscount line numbers. Echoing the diff would silently diverge from real code — violating the PRD's #1 risk ("traceable to actual hunks, never fabricate changes") — while doubling token cost and making "expand full file" + word-level highlighting impossible. Any hunk no chapter claims is bucketed into a synthetic "Unsorted changes" chapter and logged — never dropped.

05 — TWO CONTRACTS

What the agent says vs. what the UI gets

Two shapes, kept distinct. The agent contract is validated by JSON Schema on the tool call — so malformed or truncated review JSON is caught before the hook ever runs.

06 — SERVER ENDPOINTS

A tiny REST surface on an ephemeral port

One built React app fetches the review from a local server — loopback-only bind, Host-header validation, and a per-server token guard every /api/* route, plus a hardened surface: self-only CSP + nosniff on responses, size caps on the agent payload, a body limit on /api/done, and symlink-safe file reads. JSON is runtime state served over REST; highlighting runs off the main thread in a worker pool, and word-level diffing happens client-side. No per-review codegen or compile.

07 — ROADMAP

Build the viewer first; round-trip is a drop-in

The blocking hook stays from day one — so swapping the "Done" response for "approve / request-changes + annotations" later is a small change, not a rewrite. Phase 1 shipped in v0.1.0; v0.2.x added standalone multi-harness support, an executed viewer test harness, off-thread highlighting, and a hardening pass.
